# RealTimeEventLoop 계약 불일치 수정 계획

## 문제 분석

[`event_loop.py`](src/agent_trading/services/event_loop.py)가 현행 [`entities.py`](src/agent_trading/domain/entities.py), [`contracts.py`](src/agent_trading/repositories/contracts.py), [`order_manager.py`](src/agent_trading/services/order_manager.py:432)의 실제 계약과 일치하지 않습니다.

---

## 1. ExternalEventEntity 필드명 불일치 (4곳)

| event_loop.py 사용 | 실제 entities.py 필드 | 수정 방향 |
|---|---|---|
| `source=EventSource.BROKER_WS` | `source_name: str` | `source_name="broker_ws"` |
| (없음) | `published_at: datetime` **(required)** | broker event time 우선, 없으면 `ingested_at` fallback |
| `reliability_tier=SourceReliabilityTier.TIER_1` | `source_reliability_tier: str` | `source_reliability_tier=SourceReliabilityTier.T1_REGULATORY` |
| `source_id=...` | `source_event_id: str \| None` | `source_event_id=...` |
| `payload=...` | `metadata: dict[str, object]` | `metadata={...}` |
| `dedup_key=...` | `dedup_key_hash: str \| None` | `dedup_key_hash=...` |
| (없음) | `symbol: str \| None` | 설정 가능하면 추가 |

## 2. FillEventEntity 필드명/타입 불일치

| event_loop.py 사용 | 실제 entities.py | 수정 방향 |
|---|---|---|
| `fill_id=uuid4()` | `fill_event_id: UUID` | `fill_event_id=uuid4()` |
| `broker_order_id=str` | `broker_order_id: UUID` | BrokerOrderRepository로 native ID → UUID 변환 |
| `filled_quantity=int(...)` | `fill_quantity: Decimal` | `fill_quantity=Decimal(str(...))` |
| `filled_price=float(...)` | `fill_price: Decimal` | `fill_price=Decimal(str(...))` |
| `filled_at=self._parse_time(...)` | `fill_timestamp: datetime` | `fill_timestamp=_parse_time(...)` |
| `commission=None` | `fill_fee: Decimal \| None` | `fill_fee=None` |
| (없음) | `source_channel: str` **(required)** | `source_channel="websocket"` (시스템 규칙) |

**중복 방지**: `dedup_key_hash`로 ExternalEventRepository 조회 후 이미 존재하면 skip

## 3. Repository .save() → .add() (5곳)

모든 repository protocol이 `.add()` 사용. event_loop.py의 `.save()` 호출 5곳을 `.add()`로 변경.

## 4. OrderManager.transition_to() 호출 불일치

**현재 호출**: `broker_order_id=str, new_status=, reason=`
**필요한 호출**: `order=OrderRequestEntity, target_status=, reason_message=, reason_code=`

**변경 로직**:
1. WS의 `broker_order_id`(str, KIS native ID)로 `BrokerOrderRepository.get_by_native_order_id("koreainvestment", broker_order_id)` 조회
2. 조회 성공 → `OrderRepository.get(broker_order.order_request_id)` → `OrderRequestEntity`
3. 조회 실패 → ExternalEvent는 저장, OrderManager.transition_to는 skip, `reconciliation/warning` 로그만 남김
4. 얻은 Entity로 `transition_to(order_entity, target_status, reason_message=..., reason_code="WS_FILL")` 호출

**추가 주입**: `__init__`에 `broker_order_repo: BrokerOrderRepository` 파라미터 추가

## 5. Gap fill FillEvent 모델 필드명 수정

[`FillEvent`](src/agent_trading/domain/models.py:216) 모델 필드:
| 모델 필드 | event_loop.py 사용 | 수정 |
|---|---|---|
| `fill_quantity: Decimal` | `fill.filled_quantity` | `fill.fill_quantity` |
| `fill_price: Decimal` | `fill.filled_price` | `fill.fill_price` |
| `fill_timestamp: datetime` | `fill.filled_at` | `fill.fill_timestamp` |

**Gap fill account_ref 처리**: 현재 `account_ref=""` placeholder. 최소 계약 명시:
- `trigger_gap_fill()`에 `account_ref: str` 파라미터 추가 (호출자가 전달)
- `broker_order_id=""`는 그대로 유지 (전체 조회 의미)
- `from_ts` 타입 수정: `float` → `int` (REST API expect)

## 6. 통합 테스트 (`tests/services/test_event_loop_integration.py`)

### 필수 테스트 케이스

1. **WS fill notification → ExternalEventRepository.add()**
   - fill notification 메시지 수신 → 올바른 ExternalEventEntity로 `.add()` 호출 확인
   - `source_name="broker_ws"`, `source_reliability_tier="T1"`, `published_at` 존재

2. **WS fill notification → FillEventRepository.add()**
   - FillEventEntity 필드 검증: `fill_event_id`, `broker_order_id`(UUID), `fill_timestamp`, `fill_price`(Decimal), `fill_quantity`(Decimal), `source_channel="websocket"`

3. **WS fill notification → OrderManager.transition_to()**
   - BrokerOrderRepository → OrderRepository lookup 성공
   - `transition_to(order_entity, target_status, reason_message=..., reason_code="WS_FILL")` 호출 확인
   - **Partial fill 케이스**: filled_qty < order_qty → PARTIALLY_FILLED
   - **Full fill 케이스**: filled_qty >= order_qty → FILLED

4. **매핑 실패 케이스** (조건 2)
   - BrokerOrderRepository.get_by_native_order_id()가 None 반환
   - ExternalEvent는 저장되지만 OrderManager.transition_to는 호출되지 않음
   - 로그에 reconciliation/warning 기록

5. **중복 fill 방지** (조건 3)
   - 동일 dedup_key_hash로 ExternalEventRepository.find_by_dedup_key() hit
   - FillEventRepository.add() 호출 전 skip
   - 로그에 "duplicate fill" 기록

6. **Gap fill 필드 매핑**
   - Mock `adapter.get_fills()`가 `[FillEvent(...)]` 반환
   - ExternalEventEntity에 `fill_quantity`, `fill_price`, `fill_timestamp` 매핑 검증
