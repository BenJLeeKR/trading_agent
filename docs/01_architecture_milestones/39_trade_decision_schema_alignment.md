## 1. Why Now

**Plan 38** (Postgres Long-Path E2E) revealed that the `decision_id=None` workaround in E2E tests is a **symptom** of a deeper schema/entity/repository mismatch. The core system is now stable (447 tests, 0 failures), so it's time to fix the root cause rather than accumulating more workarounds.

**Problem statement**: Migration `0001_initial_schema.sql` created `trading.trade_decisions` with a `decision VARCHAR(32) NOT NULL` column (CHECK `'buy','sell','hold','reduce','exit'`). Migration `0004_expand_trade_decision.sql` added `decision_type VARCHAR(32) NOT NULL DEFAULT 'approve'` (P0 core field). The `TradeDecisionEntity` was built around `decision_type` and has **no `decision` field**. `PostgresTradeDecisionRepository.add()` INSERT includes `decision_type` but **not `decision`**. The `decision` column is entirely orphaned — no code reads or writes it.

### 2.2 Semantic Analysis: `decision` vs `decision_type`

| Property | `decision` (0001 legacy) | `decision_type` (0004 P0) |
|---|---|---|
| Type | `VARCHAR(32)` | `VARCHAR(32)` with `DecisionType` enum |
| CHECK values | `'buy','sell','hold','reduce','exit'` | None (app-layer via enum) |
| Semantics | **Trading action** (what to do with the instrument) | **AI decision type** (the verdict/type of decision) |
| Entity field | ❌ None | ✅ `DecisionType` enum |
| Used in INSERT? | ❌ No | ✅ Yes |
| Used in SELECT? | ❌ No (`row_to_entity` drops it) | ✅ Yes |
| Can map to each other? | ❌ `buy/sell` ≠ `approve/reject` | — |

**Conclusion**: These are semantically different concepts. `decision` was about the **trading action** (buy/sell/hold/reduce/exit). `decision_type` is about the **AI decision type** (approve/reject/hold/watch/exit/reduce). They cannot be meaningfully mapped to each other. The `decision` column is superseded by the combination of:
- `decision_type` (what kind of AI decision)
- `side` (trading direction: buy/sell)

### 2.3 Why `decision_id=None` Is a Workaround

There are **two distinct mechanisms** behind the workaround:

#### Mechanism 1: `decision_id` in `SubmitOrderRequest` (string UUID)
- Generated by `DecisionOrchestratorService.assemble()` at line 446-448: `decision_id = str(uuid4())` if not provided
- Used as a **correlation identifier** on `SubmitOrderRequest`
- **Not** an FK reference to `trade_decisions` table
- Setting to `None` bypasses ID generation, producing `decision_id=None` on the assembled request
- The E2E tests set `decision_id=None` to indicate "no AI agent was involved"

#### Mechanism 2: `trade_decision_id` in `OrderRequestEntity` (UUID FK)
- Stored in `order_requests.trade_decision_id` column which has a nullable FK to `trade_decisions`
- No code currently creates a `TradeDecisionEntity` → no parent row exists → FK must be `NULL`
- Tests and production code legitimately set this to `None` (valid because FK is nullable)

**Key insight**: The `decision_id=None` workaround in E2E tests is harmless for the E2E scenario (no AI agents → no decision to record). But the underlying schema constraint (`decision` column is `NOT NULL`) means that IF someone tried to create a `TradeDecisionEntity` via the Postgres repository, the INSERT would fail immediately.

## 3. Alignment Choice

### 3.1 Options Considered

| Option | Change | Risk | Entity change? | Migration needed? |
|---|---|---|---|---|
| **A. Make `decision` nullable** | `ALTER COLUMN decision DROP NOT NULL` | Minimal (no code reads it) | No | ✅ 0009 |
| **B. Add `decision` field to entity + INSERT** | Add field + map | Low (dead field) | Yes (add dead field) | No |
| **C. Drop `decision` column** | `DROP COLUMN decision` | Low (no code depends on it) | No | ✅ 0009 |
| **D. Map `decision_type → decision` in INSERT** | Derive `decision` from `side` | Medium (semantic mismatch) | No | No |

### 3.2 Recommended Choice: Option A + Migration 0009

**Make `decision` column nullable** via a new migration `0009_make_trade_decision_nullable.sql`.

Rationale:
1. **Minimal change** — single `ALTER COLUMN ... DROP NOT NULL` preserves existing data (which doesn't exist anyway)
2. **Zero entity changes** — `TradeDecisionEntity` stays exactly as-is
3. **Zero repository changes** — `add()` and `get_by_context()` continue working
4. **Zero test changes** — no test files need modification
5. **Future-proof** — if someone later creates a `TradeDecisionEntity` via Postgres, the INSERT will succeed
6. **Dead column is harmless** — standard practice in evolving schemas; can be dropped in a later cleanup

Why not Option C (drop column)?
- DROP COLUMN is a DDL lock on production tables, unnecessary risk for a harmless dead column
- The column still documents the original design intent
- Can be dropped in a future cleanup migration if desired

Why not Option B (add field)?
- Adding a semantically dead field to the entity bloats the domain model
- Violates YAGNI — no code will ever read or write this field
- The `decision` concept has been superseded by `decision_type` + `side`

Why not Option D (map)?
- Semantic mismatch: `decision` (buy/sell/hold/reduce/exit) ≠ `decision_type` (approve/reject/hold/watch/exit/reduce)
- `side` (buy/sell) ≠ `decision` (buy/sell/hold/reduce/exit) — `hold`/`reduce`/`exit` have no `side` equivalent
- Adding mapping logic creates complexity for zero benefit

### 3.3 Legacy Status of `decision` Column

The `trade_decisions.decision` column is an **orphaned legacy column** from migration 0001. It originally represented the **trading action** (buy/sell/hold/reduce/exit) — a concept that has been fully superseded by the P0 fields `decision_type` (AI decision type) and `side` (trading direction) added in migration 0004.

- **Current status**: Orphaned — no code reads or writes it. Only the `NOT NULL` constraint causes harm by blocking INSERT.
- **This plan's action**: Relax the constraint to `nullable`, acknowledging its legacy status without removing it.
- **Long-term disposition**: Deprecation candidate for future cleanup (Migration 0010+). Can be dropped once confirmed no external tooling or ad-hoc queries depend on it.

### 3.4 Decision

> **Schema → Entity alignment**: Schema's `decision` column constraint is loosened to match the entity which has no `decision` field.
>
> Justification: The entity/INSERT represent the **authoritative source** — they were deliberately designed without `decision` because `decision_type` + `side` supersede it. The schema constraint is legacy from migration 0001 and was never updated in 0004.

## 4. Migration Design

### 4.1 New Migration: `0009_make_trade_decision_nullable.sql`

```sql
BEGIN;

-- ============================================================================
-- Migration 0009: Make trade_decisions.decision nullable (Plan 39)
--
-- Problem
-- -------
-- Migration 0001 created trade_decisions.decision VARCHAR(32) NOT NULL with
-- CHECK IN ('buy','sell','hold','reduce','exit'). This column is semantically
-- superseded by decision_type (0004, P0 core field) + side (0004, P0 core field).
-- TradeDecisionEntity has no 'decision' field, and PostgresTradeDecisionRepository.add()
-- does not include it in the INSERT. The NOT NULL constraint would block any
-- future INSERT through the repository.
--
-- Fix
-- ---
-- Make the column nullable so existing INSERT statements (which omit 'decision')
-- work correctly. The column remains in the schema for backward compatibility
-- and can be dropped in a future cleanup migration if desired.
-- ============================================================================

-- Status: Orphaned legacy column superseded by decision_type + side.
--   This migration only relaxes the NOT NULL constraint so the column
--   no longer blocks INSERT. It is a deprecation candidate for future
--   removal (migration 0010+).
ALTER TABLE trading.trade_decisions
    ALTER COLUMN decision DROP NOT NULL;

COMMIT;
```

### 4.2 Migration Registration

Add `0009_make_trade_decision_nullable.sql` to `db/migrations/` directory.
