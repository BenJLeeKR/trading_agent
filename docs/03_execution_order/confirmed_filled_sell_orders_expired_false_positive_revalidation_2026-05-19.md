## Step 5: 최종 판정

### 5.1 핵심 발견: Position 감소는 확인되었으나 Cash 증가가 없음

Position snapshot에서 **분명한 수량 감소**(두산 40→10, 삼성화재 30→10)가 관찰되었으나, **Available Cash가 전혀 증가하지 않았습니다.**

이는 다음과 같은 해석이 가능합니다:

1. **Paper 환경의 특성**: Position snapshot이 broker(한국투자증권)로부터 동기화되면서 position 감소는 반영되었지만, cash 변화는 아직 반영되지 않았을 가능성
2. **Sell Order가 실제로 체결되지 않음**: Position 감소가 sell order 체결이 아닌 다른 메커니즘(예: 시스템 내부 조정, 데이터 동기화 버그)에 의한 것일 가능성
3. **Cash 동기화 지연**: Cash snapshot이 position snapshot보다 늦게 동기화되어 아직 변화가 반영되지 않았을 가능성

### 5.2 False Positive 판정

| Symbol | Position 감소 | Cash 증가 | Fill Events | 판정 |
|--------|-------------|-----------|-------------|------|
| 000150 (두산) | ✅ 30주 감소 | ❌ 변화 없음 | 0건 | **INCONCLUSIVE** |
| 000810 (삼성화재) | ✅ 20주 감소 | ❌ 변화 없음 | 0건 | **INCONCLUSIVE** |
| 000660 (SK하이닉스) | ❌ 변화 없음 | ❌ 변화 없음 | 0건 | **NOT A FALSE POSITIVE** |

**최종 결론**: Position 감소는 확인되었으나 **Cash 증가가 없고 fill_events도 없으므로**, sell order가 실제로 체결되었다고 단정할 수 없습니다. 따라서 **"EXPIRED가 false positive다"라고 판정할 수 없습니다.**

### 5.3 추가 발견: Order 상태 천이 패턴

모든 sell order는 다음과 같은 상태 천이를 보였습니다:
```
draft → validated → pending_submit → submitted → reconcile_required → expired
```

`reconcile_required` 상태는 broker로 제출 후 약 2~3분 후에 발생했으며, 이후 약 7시간 후(08:14~08:15 UTC)에 일괄적으로 `expired`로 전환되었습니다.

**`pending_submit` 상태인 4건의 주문**은 broker로 제출되지 못하고 대기 중입니다. 이 중 두산 50,000원 주문(#A)은 가격이 비정상적으로 낮아(시장가 155만원 대비) 의도치 않은低价 주문으로 보입니다.

### 5.4 권장 사항

1. **Cash snapshot 재확인 필요**: 시간이 더 지난 후 cash snapshot에 변화가 생기는지 재검증 필요
2. **Broker 측 실제 체결 확인**: 한국투자증권 API를 통해 실제 체결 여부 확인 필요
3. **Position 감소 원인 규명**: Position 감소가 실제 sell 체결에 의한 것인지, 아니면 데이터 동기화 문제인지 확인 필요
4. **`pending_submit` 주문 처리**: 4건의 미제출 주문에 대한 후속 조치 필요 (특히 두산 50,000원 주문)
