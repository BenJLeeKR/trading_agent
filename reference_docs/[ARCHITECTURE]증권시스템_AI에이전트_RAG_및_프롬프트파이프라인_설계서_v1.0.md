# 증권 AI 에이전트 아키텍처 및 프롬프트 파이프라인 설계서 (v1.0)

> 본 문서는 증권사 기간계 핵심 업무의 코딩 자동화를 목표로, AI 에이전트가 `12자리 고정 길이 표준테이블명(STD_TBL_NM)`과 `메타정보 사전`을 오차 없이 활용하여 무결성 높은 백엔드 코드와 SQL을 생성하도록 설계된 **하이브리드 RAG 및 다중 에이전트 프롬프트 파이프라인(Agentic Workflow)** 아키텍처 문서입니다.

---

## 1. 아키텍처 도입 배경 (Why this architecture?)

* **시스템의 절대적 무결성**: 전사 데이터 거버넌스를 위해 12자리 암호화된 표준명(`CORORDBST001` 등)을 도입하였으나, 이는 인간 개발자뿐만 아니라 단순 LLM에게도 높은 인지 부하와 할루시네이션(환각)을 유발할 수 있습니다.
* **해결 전략**: 개발자는 인간의 언어(논리명)로 지시하고, AI 에이전트 파이프라인은 내부적으로 하이브리드 RAG를 거쳐 물리명(`STD_TBL_NM`)으로 치환하여 코드를 작성한 뒤, 주석으로 직관성을 보장하는 양방향 번역(Translation) 아키텍처를 구현합니다.

---

## 2. 하이브리드 RAG (Hybrid RAG) 구조 설계

증권 시스템의 규칙은 텍스트(문서)와 스키마(표)가 혼합되어 있으므로 두 가지 인덱싱 방식을 병행합니다.

### 2.1 Data Ingestion (데이터 적재)
* **Vector Store (의미 기반)**
  * **저장 대상:** `물리설계 및 아키텍처 가이드`, `테이블/컬럼 명명규칙`의 텍스트 조각.
  * **목적:** "바인드 변수는 어떻게 써야 해?", "조인 표준이 뭐야?" 와 같은 추상적인 룰과 컨텍스트 검색.
* **Structured RDBMS/Graph DB (정확도 기반)**
  * **저장 대상:** `증권_META시스템_메타정보_사전`의 테이블명, 컬럼 목록.
  * **목적:** `{"논리명": "주문원장", "PHYS_TBL_NM": "TB_ORD_MST", "STD_TBL_NM": "CORORDBST001", "컬럼": [...]}` 형태의 JSON 매핑 정보 구조화.

### 2.2 Retrieval (검색 파이프라인)
* 사용자의 자연어 프롬프트가 인입되면 **Semantic Search**(문맥 파악)와 **Exact Match**(Text-to-SQL 테이블 매핑)를 동시에 수행하여, 필요한 물리 테이블 객체 정보와 코딩 규칙을 에이전트에게 100%의 정확도로 전달합니다.

---

## 3. 다중 에이전트 프롬프트 파이프라인 (Agentic Workflow)

단일 프롬프트의 한계를 극복하기 위해 역할을 세분화한 4단계 에이전트 워크플로우를 구성합니다.

### Step 1: 의도 추출 에이전트 (Intent Analyzer)
* **역할:** 개발자의 요구사항에서 '업무 도메인'과 '테이블 논리명'을 정확히 추출합니다.
* **시스템 프롬프트 예시:**
  > "당신은 증권 업무 분석가입니다. 사용자의 요청을 분석하여 필요한 증권 업무 도메인과 테이블 논리명을 JSON 배열 형태로 추출하십시오. 
  > *예: Input: '당일 매수 체결된 내역 조회 쿼리 짜줘' -> Output: ['체결내역', '주문원장']* "

### Step 2: 메타 번역 에이전트 (Meta Translator)
* **역할:** 추출된 논리 엔터티를 하이브리드 RAG에 쿼리하여 12자리 물리명(`STD_TBL_NM`)과 컬럼 명세를 확보합니다.
* **시스템 프롬프트 예시:**
  > "당신은 메타데이터 관리자입니다. RAG 시스템을 조회하여 전달받은 논리명에 1:1로 매핑되는 `STD_TBL_NM`과 필수 컬럼 목록을 가져오십시오. 만약 매핑되는 테이블이 없다면 임의로 지어내지 말고 검색을 중단(Halt)하십시오."

### Step 3: 코드 생성 에이전트 (Code Generator)
* **역할:** 확보된 스키마와 규칙을 바탕으로 실제 Java/Python 코드 및 SQL을 작성합니다.
* **시스템 프롬프트 예시:**
  > "당신은 수석 백엔드 개발자입니다. 다음의 '절대 규칙'을 준수하여 SQL 및 코드를 작성하십시오.
  > 1. RAG에서 제공한 12자리 `STD_TBL_NM`만을 FROM/JOIN 절에 사용하십시오.
  > 2. 가독성을 위해 테이블 알리아스(Alias) 선언 직후 인라인 주석으로 직관적인 논리명(`PHYS_TBL_NM`)을 반드시 병기하십시오. (예: `SELECT * FROM CORORDBST001 O /* TB_ORD_MST */`)
  > 3. 조건절에는 리터럴을 엄격히 금지하고 바인드 변수(:1, #{} 등)를 사용하십시오."

### Step 4: 품질 검증 에이전트 (Reviewer & Linter)
* **역할:** 생성된 코드가 증권 시스템 DB 물리설계 가이드(바인드 변수, ANSI JOIN, 주석 병기 등)를 위반하지 않았는지 최종 검열합니다.
* **프로세스:** 위반 사항 발견 시 Step 3의 Code Generator에게 수정(Refine) 피드백을 전달하여 코드를 재작성시킵니다.

---

## 4. 운영 및 디버깅 가이드 (개발자 인지 부하 최소화)

AI가 작성한 코드가 운영 환경에 배포된 후 장애가 발생했을 때, 개발자의 대응 속도를 높이기 위한 가이드입니다.

1. **에러 로깅 표준화:** AI가 생성하는 백엔드 애플리케이션의 에러 핸들러(Error Handler)는 `SQLException` 발생 시 에러 로그에 12자리 코드(`STD_TBL_NM`)뿐만 아니라, 항상 논리명(`PHYS_TBL_NM`)을 매핑하여 함께 출력하도록 공통 프레임워크 룰을 설정해야 합니다.
2. **코드 주석 필수 정책:** Step 3의 에이전트가 주석을 누락한 코드는 CI/CD 파이프라인에서 정적 분석(SonarQube 등)을 통해 반려되도록 룰을 추가합니다.
