## 1. DB 물리 오브젝트 명명 및 설계 규칙

### 1.1 제약조건 (Constraint) 명명 규칙
| 제약조건 유형 | Prefix | 명명 규칙 포맷 | 예시 (`CORORDBST001` 기준) |
| --- | --- | --- | --- |
| **Primary Key** | `PK_` | `PK_` + `STD_TBL_NM` | `PK_CORORDBST001` |
| **Foreign Key** | `FK_` | `FK_` + `STD_TBL_NM` + `_` + 순번(2자리) | `FK_CORORDBST001_01` |
| **Unique Key** | `UK_` | `UK_` + `STD_TBL_NM` + `_` + 순번(2자리) | `UK_CORORDBST001_01` |
| **Check** | `CK_` | `CK_` + `STD_TBL_NM` + `_` + 컬럼명 | `CK_CORORDBST001_USE_YN` |

### 1.2 인덱스 (Index) 명명 및 설계 규칙
* **개수 제한:** 트랜잭션 빈도가 높은 원장성 테이블(주문, 체결 등)은 인덱스 개수를 최대 5개 이하로 제한합니다.
* **명명 규칙:** 일반 인덱스는 `IX_` + `STD_TBL_NM` + `_` + 순번(`01`, `02`...) 포맷을 사용합니다.

---

## 2. 스토리지 및 테이블스페이스 할당 규칙

| 테이블스페이스 명칭 | 저장 대상 | 목적 및 특징 |
| --- | --- | --- |
| `TS_DAT_CORE` | 계정계 핵심 원장 데이터 (주문, 체결, 잔고) | 초고속 I/O 보장 (SSD/NVMe 전용 영역) |
| `TS_IDX_CORE` | 핵심 원장 인덱스 | 데이터 파일과 디스크 채널 분리로 경합 최소화 |
| `TS_DAT_LOB` | `CLOB`, `BLOB` (원문, 대용량 로그) | Row Chaining 방지를 위한 LOB 전용 공간 |

**동시성(Concurrency) 튜닝:** 초고도 트랜잭션 테이블은 블록 경합 방지를 위해 `INITRANS 10~20` 할당이 필수입니다.

---

## 3. 대용량 데이터 파티셔닝(Partitioning) 정책

* **Local Index 원칙:** 파티션 테이블에 생성되는 모든 인덱스는 파티션 키(`_DT`)를 포함하는 **Local Index** 생성을 원칙으로 합니다. (오래된 파티션 Drop 시 인덱스 Unusable 장애 방지 목적)

---

## 4. 데이터 라이프사이클 관리 (ILM) 및 아카이빙

* **Purge 메커니즘:** 논리삭제(`DEL_YN = 'Y'`) 데이터 및 만료 파티션은 애플리케이션 `DELETE` 대신 DBA 정기 배치(월 1회)의 파티션 `DROP` 또는 `TRUNCATE` 방식으로 제거하여 부하를 최소화합니다.

---

## 5. 표준 SQL 작성 및 성능 튜닝 가이드

* **바인드 변수(Bind Variable) 필수:** 하드 파싱(Hard Parsing)에 의한 DB 장애 방지를 위해 리터럴 텍스트 쿼리 생성을 엄격히 금지합니다.
* **대용량 배치 처리:** 단건 루프(Cursor For Loop)를 금지하고, Array Processing(`FORALL`) 또는 `INSERT /*+ APPEND */` 구문을 사용합니다.

---

## 6. DB 보안, 권한 및 접근통제 규칙

* 스키마를 오브젝트 소유자(`_OWN`), 애플리케이션 접속 계정(`_APP`), 조회 전용(`_READ`)으로 철저히 분리하고 Role 기반 접근 통제를 적용합니다.

## 7. [부록 A] 업무 데이터 이력 모델링 표준 패턴

이력 데이터의 성격과 조회 패턴에 따라 아래의 3가지 모델링 표준 패턴 중 하나를 선택하여 물리 설계에 반영해야 합니다.

### 7.1 선분이력 (Effective Dating History) 패턴
* **적용 대상:** 고객등급, 계좌상태, 수수료율 등 특정 시점의 상태를 '기간(From~To)'으로 조회해야 하는 마스터 데이터.
* **모델링 표준:** * `APLY_STRT_DT` (적용시작일자), `APLY_END_DT` (적용종료일자) 컬럼 필수 포함.
  * 최신 데이터의 종료일자는 `99991231`로 세팅하여 점조회(Point-in-time) 쿼리 성능 보장.
  * PK 구성: `기존 PK` + `APLY_STRT_DT`.

### 7.2 스냅샷 이력 (Snapshot History) 패턴
* **적용 대상:** 일마감 잔고(`TB_BAL_POS`), 예수금 일별 원장 등 특정 영업일자 기준의 전체 상태를 복사해야 하는 데이터.
* **모델링 표준:** * `BAS_DT` (기준일자) 컬럼을 PK의 가장 선두에 배치.
  * 변경 여부와 상관없이 매 배치마다 전체 활성 데이터를 복제하여 적재 (파티셔닝 필수 적용).

### 7.3 변경 델타 이력 (Delta/Transaction History) 패턴
* **적용 대상:** 원장 컬럼의 부분적 변경 내역 추적(주문 상태변경, 한도 변경 이력 등) 및 감사(Audit) 목적.
* **모델링 표준:**
  * 기존 테이블명 뒤에 `_HIST` (또는 `HST`) Suffix 사용.
  * `HIST_SEQ` (이력순번), `BEF_VAL` (변경전값), `AFT_VAL` (변경후값), `CHG_RSN` (변경사유) 컬럼 구성.
  * 트리거(Trigger) 방식은 I/O 경합을 유발하므로 금지하며, 애플리케이션 또는 CDC 솔루션 기반 적재 원칙.

## 8. [부록 B] 성능 지향 역정규화 및 데이터 정합성 통제 기준

증권 시스템의 초고속(Low-Latency) 매매 처리를 위해 일부 정규화를 파기하는 **역정규화(Denormalization)**를 허용하되, 아래의 통제 기준을 엄수해야 합니다.

### 8.1 역정규화 허용 대상 및 기준
1. **실시간 산출 병목 컬럼:** * 주문원장의 `EXEC_ACML_QTY` (체결누계수량), `UNEXEC_QTY` (미체결수량).
   * 예수금잔고의 `ORD_ABLE_AMT` (실시간주문가능금액).
   * 매번 하위 체결 트랜잭션을 `SUM()` 연산할 경우 호가 엔진 응답 속도가 저하되므로 부모 마스터 원장에 중복 저장(역정규화)을 허용합니다.
2. **코드명/명칭 컬럼 중복 금지:** * 조인을 피하기 위해 `ISU_NM`(종목명)이나 `CUST_NM`(고객명)을 원장에 역정규화하는 것은 엄격히 금지합니다. (조회용 캐시나 뷰로 해결).

### 8.2 역정규화 데이터 정합성(Integrity) 보장 대책
역정규화된 수치 데이터는 시스템 오류나 동시성 경합 시 원본 트랜잭션과 불일치가 발생할 위험이 있습니다.
* **비동기 정합성 체크:** 야간 일마감 배치 수행 전, `주문수량 = 체결누계 + 미체결 + 취소` 산식이 1원/1주 오차 없이 맞는지 교차 검증(`BAT_ORD_RECON`)을 필수로 수행해야 합니다.
* **보정 전표 프로세스:** 불일치 발생 시 원장을 직접 `UPDATE` 하지 않고, 차이를 맞추기 위한 '시스템 보정 전표/주문'을 발생시켜 감사 추적성을 확보해야 합니다.
