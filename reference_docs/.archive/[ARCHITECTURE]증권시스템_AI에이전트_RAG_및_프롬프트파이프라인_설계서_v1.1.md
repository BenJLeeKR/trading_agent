# 증권 AI 에이전트 아키텍처 및 프롬프트 파이프라인 설계서 (v1.1)

> 본 문서는 증권사 기간계 핵심 업무의 코딩 자동화를 목표로 작성되었습니다. AI 에이전트가 `12자리 고정 길이 표준테이블명(STD_TBL_NM)`과 `메타정보 사전`을 오차 없이 활용하여 무결성 높은 백엔드 코드와 SQL을 생성하도록 설계된 **하이브리드 RAG 및 다중 에이전트 프롬프트 파이프라인(Agentic Workflow)의 초상세 아키텍처 가이드**입니다.

---

## 1. 아키텍처 도입 배경 및 핵심 과제

### 1.1 왜 하이브리드 파이프라인인가?
* **문제점:** 전사 데이터 거버넌스를 위해 도입된 12자리 암호화 표준명(예: `CORORDBST001`)은 관리에는 용이하지만, LLM(대형 언어 모델)에게는 심각한 인지 부하와 할루시네이션(환각)을 유발합니다. AI가 쿼리를 작성할 때 테이블명을 헷갈려 잘못된 조인을 수행할 확률이 매우 높습니다.
* **해결책:** 개발자는 인간의 언어(논리명)로 지시하고, AI 파이프라인은 내부적으로 **하이브리드 RAG(의미 검색 + 스키마 맵핑)**를 거쳐 물리명(`STD_TBL_NM`)으로 치환하여 코드를 작성합니다. 결과 코드에는 주석으로 논리명을 병기하여 운영자의 가독성까지 보장합니다.

---

## 2. 하이브리드 RAG (Hybrid RAG) 상세 설계 및 데이터 모델

증권 시스템의 복잡한 규칙과 스키마를 에이전트가 정확히 인식하려면 텍스트와 정형 데이터 인덱싱을 병행해야 합니다.

### 2.1 Data Ingestion (데이터 적재 전략)
1. **Vector Store (의미 및 룰 기반 인덱싱)**
   * **저장 대상:** `[RULES]증권시스템_DB물리설계_가이드`, `[RULES]컬럼명명규칙` 등의 텍스트 문서 조각(Chunk).
   * **활용:** "바인드 변수 처리 기준", "이력 데이터 모델링 패턴" 등 개발 룰과 컨텍스트를 검색할 때 사용됩니다.
2. **Structured DB / Knowledge Graph (정확도 기반 맵핑)**
   * **저장 대상:** `증권_META시스템_메타정보_사전`의 테이블명, 컬럼 목록, 도메인.
   * **자료 구조 (JSON 예시):**
     ```json
     {
       "logical_name": "주문원장",
       "phys_tbl_nm": "TB_ORD_MST",
       "std_tbl_nm": "CORORDBST001",
       "primary_keys": ["ORD_DT", "ORD_NO"],
       "columns": [
         {"col_nm": "ISU_CD", "type": "VARCHAR2(12)", "desc": "종목코드"},
         {"col_nm": "ORD_QTY", "type": "NUMBER(18,6)", "desc": "주문수량"}
       ]
     }
     ```

### 2.2 Retrieval Algorithm (검색 및 주입 로직)
사용자의 프롬프트가 입력되면 두 갈래로 검색이 병렬 실행됩니다.
* **경로 A (Semantic Search):** 자연어에서 업무 의도를 파악하여 관련 설계 규칙(Rule)을 페치합니다.
* **경로 B (Exact Match Text-to-SQL):** 추출된 논리명(`주문원장`)을 Key로 하여 Structured DB에서 정확도 100%의 `CORORDBST001` 맵핑 데이터와 스키마를 추출합니다.

## 3. 다중 에이전트 프롬프트 파이프라인 (4-Step Agentic Workflow)

단일 프롬프트의 한계를 극복하고 자기 검증(Self-Correction)이 가능하도록 4단계 에이전트 릴레이로 구성합니다.

### Step 1: 의도 추출 에이전트 (Intent Analyzer)
* **역할:** 개발자의 요구사항에서 '업무 도메인'과 '테이블 논리명'을 정확히 추출하여 배열화합니다.
* **Input:** `"알테오젠 당일 매수 체결된 내역과 원주문 상태를 조인해서 가져오는 쿼리 짜줘."`
* **시스템 프롬프트 지시:**
  > "당신은 증권 업무 분석가입니다. 사용자의 요청을 분석하여 추출된 테이블 논리명을 JSON 배열로 출력하십시오. 업무 외적인 잡담은 제외하십시오."
* **Output:** `["체결내역", "주문원장"]`

### Step 2: 메타 번역 에이전트 (Meta Translator)
* **역할:** 추출된 논리 엔터티를 하이브리드 RAG에 쿼리하여 12자리 물리명(`STD_TBL_NM`)과 DDL 컬럼 명세를 확보합니다.
* **시스템 프롬프트 지시:**
  > "당신은 메타데이터 관리자입니다. 제공된 논리명 배열을 RAG Knowledge Base에서 검색하여 1:1로 매핑되는 `STD_TBL_NM`과 조인에 필요한 PK 컬럼 목록을 가져오십시오. **주의:** 매핑되는 테이블이 없다면 절대로 임의의 12자리 코드를 지어내지 말고, 빈 배열을 반환하며 프로세스를 중단(Halt)하십시오."

### Step 3: 코드 생성 에이전트 (Code Generator)
* **역할:** 확보된 스키마와 물리 설계 룰을 바탕으로 실제 애플리케이션 코드(Java/Python)와 SQL을 작성합니다.
* **시스템 프롬프트 지시:**
  > "당신은 수석 백엔드 개발자입니다. 다음의 '절대 규칙'을 준수하여 SQL을 작성하십시오.
  > 1. RAG에서 제공한 12자리 `STD_TBL_NM`만을 FROM/JOIN 절에 사용하십시오.
  > 2. 가독성을 위해 테이블 알리아스(Alias) 선언 직후 인라인 주석으로 직관적인 논리명(`PHYS_TBL_NM`)을 반드시 병기하십시오. 
  >    *올바른 예: `SELECT * FROM CORORDBST001 O /* TB_ORD_MST */`*
  > 3. 조건절에는 리터럴을 엄격히 금지하고 바인드 변수(:1, #{} 등)를 사용하십시오."

### Step 4: 품질 검증 에이전트 (Reviewer & Linter)
* **역할:** 생성된 코드가 `DB 물리설계 가이드`를 위반하지 않았는지 최종 검열합니다.
* **검열 체크리스트 루프:**
  1. FROM 절에 12자리 고정 규칙(`^[A-Z0-9]{12}$`) 외의 이름이 쓰였는가?
  2. 리터럴 값 검색이 바인드 변수로 처리되었는가?
  3. 조인 시 ANSI 문법을 사용하였는가?
* **피드백 액션:** 위반 사항 발견 시, 코드를 즉각 Step 3(Code Generator)으로 돌려보내며 수정(Refine)을 지시하는 프롬프트를 발송합니다.

## 4. 운영 및 디버깅 가이드 (개발자 인지 부하 제로화 정책)

AI가 완벽하게 코드를 작성하여 배포했더라도, 운영 중 에러 로그 확인 시 개발자의 피로도를 낮춰야 합니다.

1. **에러 로깅 표준 프레임워크 연계:** * AI가 생성하는 백엔드 애플리케이션의 에러 핸들러(ExceptionHandler)는 예외 발생 시 로그에 `CORORDBST001`이라는 식별자만 남기지 않도록 합니다. 반드시 런타임 캐시를 참조하여 논리명(`TB_ORD_MST`)을 에러 스택트레이스에 병기하여 출력하도록 로깅 모듈 규칙을 에이전트에게 강제합니다.
2. **CI/CD 파이프라인 정적 분석:** * 주석 누락(`/* 물리명 */`) 또는 바인드 변수 누락 코드는 SonarQube 등 정적 분석 단계에서 정규식 패턴 검사로 즉각 빌드 반려되도록 CI 파이프라인을 구축해야 합니다.
