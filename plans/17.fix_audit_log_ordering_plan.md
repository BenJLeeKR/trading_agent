# Audit Log Ordering Fix Plan

## Problem

`test_list_by_correlation_id_ordered` intermittently fails because
[`PostgresAuditLogRepository.list_by_correlation_id`](src/agent_trading/repositories/postgres/audit_logs.py:55)
uses `ORDER BY created_at` as the sole sort criterion. When multiple rows
share the **same** `created_at` timestamp (as the test does explicitly at
[`test_postgres_audit_logs.py:70`](tests/repositories/test_postgres_audit_logs.py:70)),
PostgreSQL returns rows in arbitrary physical storage order.

## Fix

Add an `audit_log_seq BIGSERIAL` column to `trading.audit_logs` and use it
as a secondary sort key: `ORDER BY created_at, audit_log_seq`.

- `BIGSERIAL` is auto-incrementing per row — guarantees insertion order.
- Application code sets `created_at` explicitly; `audit_log_seq` is purely
  an infrastructure concern for deterministic ordering.

## Additional conditions

### 1. Migration SQL (`db/migrations/0007_add_audit_log_seq.sql`)

```sql
-- Migration 0007: Add audit_log_seq to audit_logs for deterministic ordering
--
-- The BIGSERIAL column provides an auto-incrementing sequence number
-- that guarantees strict insertion order, serving as a tie-breaker
-- when multiple rows share the same created_at timestamp.

BEGIN;

-- Add the sequence column (BIGSERIAL auto-creates a sequence)
ALTER TABLE trading.audit_logs ADD COLUMN audit_log_seq BIGSERIAL;

-- Backfill existing rows with sequential values (ordered by created_at, audit_log_id)
-- This ensures existing data gets deterministic audit_log_seq values.
-- The subquery assigns row_number() based on created_at (NULLs last) + audit_log_id.
UPDATE trading.audit_logs AS t
SET audit_log_seq = s.seq
FROM (
    SELECT audit_log_id,
           row_number() OVER (ORDER BY created_at NULLS LAST, audit_log_id) AS seq
    FROM trading.audit_logs
) s
WHERE t.audit_log_id = s.audit_log_id;

-- Drop the old single-column index; replace with composite for WHERE + ORDER BY
DROP INDEX IF EXISTS trading.idx_audit_logs_correlation_id;

CREATE INDEX IF NOT EXISTS idx_audit_logs_correlation_id_created_seq
    ON trading.audit_logs (correlation_id, created_at DESC, audit_log_seq DESC);

COMMIT;
```

### 2. All time-based query paths

The only time-based query in [`PostgresAuditLogRepository`](src/agent_trading/repositories/postgres/audit_logs.py:51)
is `list_by_correlation_id`.  No additional query paths exist in the protocol
or implementation — the tie-breaker is applied to the sole sorted query.

### 3. Composite index

The new index `idx_audit_logs_correlation_id_created_seq` on
`(correlation_id, created_at DESC, audit_log_seq DESC)` serves both the
`WHERE correlation_id = $1` filter and the `ORDER BY created_at, audit_log_seq`
sort in a single index scan, avoiding an in-memory sort.

### 4. Existing data safety

- `ALTER TABLE ... ADD COLUMN col BIGSERIAL` auto-fills new rows with `nextval()`
- The explicit `UPDATE` with `row_number() OVER (ORDER BY created_at, audit_log_id)`
  guarantees a deterministic sequence for existing rows, ordered chronologically
  with `audit_log_id` as tie-breaker (UUID comparison is deterministic).
- This is a one-time backfill; new inserts will use the sequence naturally.
