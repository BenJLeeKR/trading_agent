## 3. 주문 상태 전이표 (Order State Machine)

### 3.1 상태 정의

```text
상태              | 설명
DRAFT             | 주문 요청 생성, 검증 전
VALIDATED         | 기본 검증 통과
PENDING_SUBMIT    | 브로커 제출 대기 (orchestrator가 처리 중)
SUBMITTED         | 브로커에 전송 성공
ACKNOWLEDGED      | 브로커가 주문 접수 확인
PARTIALLY_FILLED  | 일부 체결됨
FILLED            | 전량 체결됨
CANCEL_PENDING    | 취소 요청 전송됨 (브로커 응답 대기)
CANCELLED         | 취소 완료
REJECTED          | 브로커가 주문 거부
EXPIRED           | 장 종료 등으로 자동 만료
RECONCILE_REQUIRED| 브로커 상태 불확실, 정합성 복구 필요
```

### 3.2 허용 전이표

| 현재 상태 → 다음 상태 | 트리거 | 조건 |
|----------------------|--------|------|
| `DRAFT` → `VALIDATED` | validate() 통과 | 수량 > 0, 필수 필드 존재 |
| `VALIDATED` → `PENDING_SUBMIT` | prepare_submit() | idempotency 중복 없음 |
| `PENDING_SUBMIT` → `SUBMITTED` | transport 성공 | HTTP 200/202 |
| `SUBMITTED` → `ACKNOWLEDGED` | broker 접수 응답 | broker_order_id 존재 |
| `SUBMITTED` → `RECONCILE_REQUIRED` | submit timeout/unknown | 응답 없음 or 모호 |
| `ACKNOWLEDGED` → `PARTIALLY_FILLED` | fill 이벤트 수신 | fill_qty < requested_qty |
| `ACKNOWLEDGED` → `FILLED` | fill 이벤트 수신 | fill_qty == requested_qty |
| `ACKNOWLEDGED` → `CANCELLED` | cancel 응답 성공 | broker 취소 확인 |
| `ACKNOWLEDGED` → `REJECTED` | broker 거부 응답 | 명시적 거부 코드 |
| `ACKNOWLEDGED` → `RECONCILE_REQUIRED` | 상태 불확실 | 조회 결과 모호 |
| `PARTIALLY_FILLED` → `FILLED` | 추가 fill | 누적 fill == requested_qty |
| `PARTIALLY_FILLED` → `CANCEL_PENDING` | cancel 요청 | 잔량 취소 |
| `PARTIALLY_FILLED` → `RECONCILE_REQUIRED` | 장시간 무응답 | 설정 시간 초과 |
| `CANCEL_PENDING` → `CANCELLED` | broker 취소 확인 | |
| `CANCEL_PENDING` → `RECONCILE_REQUIRED` | 취소 응답 불확실 | |
| `*` → `RECONCILE_REQUIRED` | 정합성 불일치 | broker-내부 상태 불일치 |
| `RECONCILE_REQUIRED` → `ACKNOWLEDGED` | reconcile 결과 정상 | |
| `RECONCILE_REQUIRED` → `FILLED` | reconcile 결과 채결 확인 | |
| `RECONCILE_REQUIRED` → `CANCELLED` | reconcile 결과 취소 확인 | |
| `RECONCILE_REQUIRED` → `REJECTED` | reconcile 결과 거부 확인 | |
| `RECONCILE_REQUIRED` → `EXPIRED` | reconcile 결과 만료 확인 | |

### 3.3 금지 전이

| 금지 전이 | 이유 |
|-----------|------|
| `FILLED` → `CANCELLED` | 체결 완료 후 취소 불가 |
| `FILLED` → `REJECTED` | 체결 완료 후 거부 불가 |
| `FILLED` → `PARTIALLY_FILLED` | 역전이 불가 |
| `CANCELLED` → `FILLED` | 취소 완료 후 체결 불가 |
| `REJECTED` → `ACKNOWLEDGED` | 거부 완료 후 접수 불가 |
| `EXPIRED` → `SUBMITTED` | 만료 후 재전송 불가 |
| `DRAFT` → `SUBMITTED` | 검증 생략 금지 |
| `VALIDATED` → `FILLED` | 전송 없이 체결 불가 |

### 3.4 최종 상태 (Terminal States)

`FILLED`, `CANCELLED`, `REJECTED`, `EXPIRED` — 이 상태에서는 더 이상 전이하지 않는다.

## 4. DDL-Entity 매핑 규칙

### 4.1 기본 매핑 규칙

| DB 타입 | Python 타입 | 변환 규칙 |
|---------|------------|-----------|
| `UUID` | `UUID` (from `uuid`) | `str → UUID` 변환, DB `gen_random_uuid()` |
| `VARCHAR(n)` / `TEXT` | `str` | 직접 매핑 |
| `NUMERIC(p,s)` | `Decimal` | `str → Decimal` (KIS 문자열 숫자 대비) |
| `TIMESTAMPTZ` | `datetime` (timezone-aware) | UTC 기준 저장/조회 |
| `JSONB` | `dict[str, object]` | `json.dumps()` / `json.loads()` |
| `BOOLEAN` | `bool` | 직접 매핑 |
| `BIGINT` | `int` | 직접 매핑 |
| `DATE` | `date` | 직접 매핑 |

### 4.2 Enum 저장 규칙

- DB: `VARCHAR` + `CHECK` 제약 조건
- Python: `str` 상속 `Enum`
- 변환: `entity.status.value` → DB 저장, DB 값 → `OrderStatus(value)` 복원

### 4.3 snake_case 변환 규칙

DB 컬럼명은 snake_case, Python 필드명은 동일하게 유지한다.
`domain/entities.py`의 dataclass 필드명 = DDL 컬럼명 = SQL 쿼리 컬럼명

예외: Python 키워드 충돌 시에만 별도 매핑 (현재 없음)

### 4.4 Nullable 처리

- DB 컬럼에 `NOT NULL`이 없는 경우 → Python `Optional[T]`
- 모든 `created_at`, `updated_at`은 DB 기본값(`DEFAULT NOW()`) 사용
- INSERT 시 DB 기본값에 의존하는 컬럼은 생략 가능

### 4.5 Row Mapper 공통 함수

```python
def row_to_entity(row: asyncpg.Record, entity_class: type[T]) -> T:
    """asyncpg row → dataclass entity 변환."""
    fields = {k: v for k, v in dict(row).items() if k in entity_class.__dataclass_fields__}
    return entity_class(**fields)
```

## 5. OrderManager 책임과 검증 범위

### 5.1 OrderManager 책임

OrderManager는 **판단자가 아니라 실행 오케스트레이터**다.

```
입력: SubmitOrderRequest (검증된 주문 의도)
책임:
  1. client_order_id 생성 (or 수신)
  2. idempotency_key 생성
  3. idempotency 사전 조회 (중복 확인)
  4. 주문 엔티티 생성 (DRAFT)
  5. 기본 필드 검증 (수량, 필수 필드) → VALIDATED
  6. PENDING_SUBMIT 전이
  7. repository 저장
  8. audit log 생성
  9. (향후) BrokerRouter 호출 위임
 10. broker 응답에 따른 상태 전이
```

### 5.2 최소 검증 범위 (Phase 1)

| 검증 항목 | 내용 | 실패 시 |
|-----------|------|---------|
| 수량 검증 | `requested_quantity > 0` | ValueError |
| 필수 필드 검증 | account_id, instrument_id, side, order_type 존재 | ValueError |
| ID 중복 검증 | `client_order_id` / `idempotency_key` UNIQUE 위반 | DuplicateOrderError |
| 상태 전이 검증 | 허용 전이표 기준 금지 전이 차단 | InvalidStateTransitionError |

### 5.3 OrderManager가 하지 않는 것

- AI 판단
- 리스크 평가
- Hard Guardrail 한도 검증
- 브로커 capability 확인
- 포트폴리오 비중 계산
- 체결/잔고 reconciliation

## 6. Idempotency 처리 규칙

### 6.1 이중 방어 구조

```
Layer 1: 애플리케이션 레벨 사전 조회
  - 동일 client_order_id로 기존 주문 존재 시 → 기존 주문 반환 (409 Conflict)
  - 동일 idempotency_key로 기존 주문 존재 시 → 기존 주문 반환 (409 Conflict)

Layer 2: DB UNIQUE 제약
  - DDL: UNIQUE (client_order_id), UNIQUE (idempotency_key)
  - UNIQUE 위반 시 → IntegrityError catch → DuplicateOrderError 변환
```

### 6.2 Idempotency Key 생성 규칙

```python
# 구성 요소
components = [
    environment,        # paper / live
    account_id,         # 계좌 식별자
    trading_day,        # 거래일 YYYYMMDD
    strategy_id,        # 전략 식별자
    sequence,           # 전략 내 순차 번호
]
idempotency_key = hashlib.sha256(":".join(components).encode()).hexdigest()
```

### 6.3 재시도 정책

- submit 실패 시 **자동 재시도 금지**
- 불확실 상태 → `RECONCILE_REQUIRED` 전이 → operator 판단 또는 reconcile worker 처리
- 같은 의도의 재시도는 같은 `client_order_id` 사용 → idempotency가 중복 차단

## 7. Audit Log 필수 필드와 기록 시점

### 7.1 필수 필드

| 필드 | 타입 | 설명 |
|------|------|------|
| `actor_type` | `system / operator / agent` | 행위 주체 유형 |
| `actor_id` | str | 행위 주체 식별자 |
| `action` | str | 수행된 액션 (예: `order.create`, `order.status_change`) |
| `target_entity_type` | str | 대상 엔티티 유형 (예: `order_request`) |
| `target_entity_id` | str | 대상 엔티티 ID |
| `before_json` | dict or None | 변경 전 상태 |
| `after_json` | dict or None | 변경 후 상태 |
| `correlation_id` | str | 요청 추적 ID |
| `metadata` | dict | 추가 메타데이터 |
| `created_at` | datetime | 자동 설정 |

### 7.2 기록 시점

| 이벤트 | action 값 | before | after |
|--------|-----------|--------|-------|
| 주문 생성 (DRAFT) | `order.create` | `None` | 주문 초기 상태 |
| 상태 전이 | `order.status_change` | 이전 상태 | 새 상태 |
| 중복 주문 차단 | `order.duplicate_blocked` | 대상 주문 상태 | 차단 로그 |
| 검증 실패 | `order.validation_failed` | `None` | 오류 내용 |

### 7.3 마스킹 규칙

- 계좌번호: `account_masked` 필드만 저장, 원본 계좌번호 미저장
- 토큰/Appkey/Appsecret: 절대 로그 금지
- Raw payload: `raw_payload_uri`로 외부 참조, 본문 미저장
- 민감정보 필드 마스킹: 정규식 치환 후 저장

## 9. 구현 순서 (7단계)

### Step 1: DB foundation
- `db/connection.py` — asyncpg `create_pool()`, get_pool(), close_pool()
- `db/transaction.py` — `TransactionManager` context manager (begin/commit/rollback)
- `db/row_mapper.py` — `row_to_entity()` 공통 변환 함수
- `db/migrations/run.py` — DDL 파일 읽어서 실행

### Step 2: PostgreSQL repositories (4개)
- `repositories/postgres/clients.py`
- `repositories/postgres/accounts.py`
- `repositories/postgres/instruments.py`
- `repositories/postgres/orders.py`

각 repository는 `repositories/contracts.py`의 Protocol을 만족시킨다.

### Step 3: UnitOfWork 구현
- `repositories/postgres_uow.py` — asyncpg transaction 기반

### Step 4: Bootstrap 조립
- `repositories/postgres/bootstrap.py` — postgres 기반 `RepositoryContainer` 생성
- `runtime/bootstrap.py` — in-memory 유지, postgres 경로 추가 (전환 가능 구조)

### Step 5: OrderManager 확장
- `services/order_manager.py` — 상태 전이, idempotency 검사, audit log, repository 연동

### Step 6: 검증 코드
- repository contract test (각 CRUD)
- order 상태 전이 test (허용/금지 전이)
- idempotency test (중복 차단)
- paper smoke test (seed → insert → query)

### Step 7: 최종 문서화
- `plans/mvp_milestone1_completion_report.md` — 완료 보고서
