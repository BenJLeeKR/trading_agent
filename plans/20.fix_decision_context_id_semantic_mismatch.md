# decision_context_id 의미 정리: Payload vs Storage ID

## 문제

현재 [`recorder.py`](src/agent_trading/services/ai_agents/recorder.py:99)는 `decision_context_id=None`일 때 하나의 synthetic UUID를 생성하여 **entity의 storage ID**와 **structured_output의 payload** 양쪽에 동일하게 사용합니다.

```
decision_context_id=None
    ↓
resolved_ctx_id = uuid4()      ← 한 번 생성
    ↓
entity.decision_context_id = resolved_ctx_id          (storage ID)
structured_output["decision_context_id"] = str(...)    (payload — 왜곡!)
```

이는 두 의미를 혼동합니다:

| 항목 | Storage 의미 (AgentRunEntity) | Payload 의미 (structured_output_json) |
|------|-------------------------------|----------------------------------------|
| 역할 | DB 조회/그룹핑용 내부 식별자 | "이 run이 어떤 decision context에 속하는가" |
| null 의미 | 없음 — UUID 필수 | 있음 — "어떤 context에도 속하지 않음" |
| synthetic UUID | `decision_context_id or uuid4()` 로 적절 | ❌ 여기에 들어가면 downstream 해석 오류 |

## 해결 방향

`structured_output_json["decision_context_id"]`는 payload 의미를 유지해 **null을 허용**합니다.
`AgentRunEntity.decision_context_id`는 내부 storage를 위해 synthetic UUID를 사용할 수 있습니다.

## 변경 사항

### 1. [`recorder.py`](src/agent_trading/services/ai_agents/recorder.py) — 의미 분리

현재 코드 (문제):
```python
resolved_ctx_id: UUID = decision_context_id or uuid4()
stored_ctx_id = output_dict.get("decision_context_id")
expected_ctx_id = str(resolved_ctx_id)
if stored_ctx_id != expected_ctx_id:
    ...
    output_dict["decision_context_id"] = expected_ctx_id  # ← synthetic UUID가 payload에 들어감

run = AgentRunEntity(decision_context_id=resolved_ctx_id, ...)
```

수정 코드:
```python
# ── 내부 storage ID (entity) ──
resolved_entity_ctx_id: UUID = decision_context_id or uuid4()

# ── payload (structured_output): 의미를 유지 ──
if decision_context_id is not None:
    # 실제 context가 있을 때만 일치 검증 후 payload에 기록
    expected = str(decision_context_id)
    stored = output_dict.get("decision_context_id")
    if stored != expected:
        if stored is not None:
            logger.warning(...)
        output_dict["decision_context_id"] = expected
else:
    # context가 없으면 payload는 null로 유지 (synthetic UUID 금지)
    stored = output_dict.get("decision_context_id")
    if stored is not None:
        logger.warning(
            "decision_context_id set in structured_output but no "
            "decision context provided — removing from payload"
        )
        output_dict.pop("decision_context_id", None)

run = AgentRunEntity(decision_context_id=resolved_entity_ctx_id, ...)
```

결과 행동:
```
decision_context_id=None
    → entity.decision_context_id = synthetic UUID    (storage OK)
    → structured_output["decision_context_id"] = None  (payload null)

decision_context_id=UUID("abc-...")
    → entity.decision_context_id = UUID("abc-...")
    → structured_output["decision_context_id"] = "abc-..."
```

### 2. [`test_orchestrator_agents.py`](tests/services/ai_agents/test_orchestrator_agents.py:309) — 테스트 이원화

**기존 테스트** `test_structured_output_contains_decision_context_id`:
- `assemble(sample_request)` — `decision_context_id` 없이 호출
- 검증: 모든 run의 `structured_output["decision_context_id"]`가 `None`인지 확인
- entity의 synthetic UUID와는 비교하지 않음 (의미 분리 반영)

**신규 테스트** `test_structured_output_contains_decision_context_id_when_provided`:
- `assemble(sample_request, decision_context_id=ctx_id)` — context ID 제공
- 검증: 모든 run의 `structured_output["decision_context_id"]`가 `str(ctx_id)`와 일치

**영향 없음**: `test_assemble_with_decision_context_id` — 이 테스트는 `intent.decision_context_id`만 검증하므로 변경 불필요

### 3. [`entities.py`](src/agent_trading/domain/entities.py:160) — 변경 불필요

`AgentRunEntity.decision_context_id: UUID` (non-nullable) 유지.
synthetic UUID가 storage 레벨에서 정상 동작함.
