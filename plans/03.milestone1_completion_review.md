## 3. 동시성 / 경합 리스크 정리

### 3.1 동일 idempotency_key 동시 요청 시 기대 동작

```
요청 A (tx1)                    요청 B (tx2)
  │                               │
  ├─ app-level pre-check          ├─ app-level pre-check
  │  get_by_client_order_id()     │  get_by_client_order_id()
  │  → None (아직 없음)           │  → None (아직 없음)
  │                               │
  ├─ INSERT (tx1 commit)          ├─ INSERT (tx2 commit)
  │  → 성공                       │  → UniqueViolationError ❗
  │                               │
  └─ return OK                    └─ DuplicateClientOrderIdError raise
```

**핵심:** app-level pre-check(Layer 1)는 동시 요청을 완전히 막을 수 없음. **DB UNIQUE 제약(Layer 2)이 최종 방어선** 역할.

### 3.2 UNIQUE 위반 예외 처리 방식

[`PostgresOrderRepository.add()`](src/agent_trading/repositories/postgres/orders.py:63)에서 `asyncpg.UniqueViolationError`를 catch하여 구체적 예외로 변환:

```python
except asyncpg.UniqueViolationError as exc:
    constraint = exc.constraint_name
    if constraint == "uq_order_requests_client_order_id":
        raise DuplicateClientOrderIdError(order.client_order_id) from exc
    if constraint == "uq_order_requests_idempotency_key":
        raise DuplicateIdempotencyKeyError(order.idempotency_key) from exc
    raise
```

| 예외 클래스 | 발생 조건 | 사용자 메시지 |
|-------------|-----------|---------------|
| `DuplicateClientOrderIdError` | 동일 `client_order_id` INSERT | `"client_order_id '<value>' already exists"` |
| `DuplicateIdempotencyKeyError` | 동일 `idempotency_key` INSERT | `"idempotency_key '<value>' already exists"` |

### 3.3 재시도 시나리오에서 상태 일관성

| 시나리오 | 위험 | 방어 방법 |
|----------|------|-----------|
| 동일 요청 재시도 (멱등) | 중복 주문 생성 | Layer 1: app-level pre-check (`get_by_client_order_id`) → 이미 존재하면 `DuplicateOrderError` |
| | DB 제약 우회 | Layer 2: `uq_order_requests_client_order_id` UNIQUE 제약 |
| 네트워크 타임아웃 후 재시도 | 첫 요청은 성공했으나 클라이언트가 실패로 인지 | `client_order_id` 기반 조회로 기존 주문 반환 (멱등성 보장) |
| 상태 전이 중 재시도 | 이중 상태 변경 | `update_status`는 단순 UPDATE이므로 동일 값 재적용은 safe (멱등) |
| 부분 체결 후 재시도 | `PARTIALLY_FILLED` → `FILLED` 전이 중복 | 상태 전이 검증(`_validate_transition`)에서 `FILLED`는 terminal state이므로 재전이 불가 |

### 3.4 트랜잭션 격리 수준

- `asyncpg` 기본 격리 수준: `READ COMMITTED`
- 동시성 제어는 **낙관적** 방식 (별도의 `SELECT ... FOR UPDATE`나 `LOCK TABLE` 없음)
- `OrderManager.create_order()`와 `transition_to()`는 각각 단일 SQL로 처리되므로 `READ COMMITTED`에서도 안전

### 3.5 잔여 리스크

| 리스크 | 영향 | 우선순위 |
|--------|------|----------|
| `transition_to()` 호출 전 order가 변경된 경우 | 낡은 상태 기준으로 전이 검증 | 중간 — `updated_at` 낙관적 Lock 고려 필요 |
| 동시 `create_order` + `transition_to` | 없는 order에 전이 시도 → `UPDATE 0 rows` | 낮음 — `UPDATE` 결과 rows 확인 로직 없음 |
| `InMemoryAuditLogRepository` 사용 중 | audit log 유실 가능 | 낮음 — Postgres 구현 전까지 허용 |

## 4. 상태 전이표 (State Transition Table)

### 4.1 허용 전이 (Allowed Transitions)

`_ALLOWED_TRANSITIONS` ([`src/agent_trading/services/order_manager.py:52`](src/agent_trading/services/order_manager.py:52))에 정의:

```
DRAFT ──────────────► VALIDATED
                       │
                       ▼
                  PENDING_SUBMIT ──────► SUBMITTED
                       │                    │
                       ▼                    ▼
                  RECONCILE_REQUIRED    ACKNOWLEDGED ──► PARTIALLY_FILLED ──► FILLED (terminal)
                                             │                │
                                             ├──► CANCELLED    ├──► CANCEL_PENDING ──► CANCELLED
                                             ├──► REJECTED     │
                                             └──► RECONCILE    └──► RECONCILE_REQUIRED
                                                    │
                                                    ├──► ACKNOWLEDGED
                                                    ├──► FILLED
                                                    ├──► CANCELLED
                                                    ├──► REJECTED
                                                    └──► EXPIRED
```

### 4.2 금지 전이 (Forbidden Transitions)

`_validate_transition()` ([`src/agent_trading/services/order_manager.py:95`](src/agent_trading/services/order_manager.py:95))에서 검증:

| from → to | 금지 이유 |
|-----------|-----------|
| `DRAFT → SUBMITTED` | 검증 단계 생략 |
| `VALIDATED → FILLED` | 브로커 제출 생략 |
| `FILLED → anything` | Terminal state — 변경 불가 |
| `CANCELLED → anything` | Terminal state — 변경 불가 |
| `REJECTED → anything` | Terminal state — 변경 불가 |
| `EXPIRED → anything` | Terminal state — 변경 불가 |

### 4.3 최종 상태 (Terminal States)

```python
_TERMINAL_STATES = frozenset({
    OrderStatus.FILLED,
    OrderStatus.CANCELLED,
    OrderStatus.REJECTED,
    OrderStatus.EXPIRED,
})
```

Terminal state에 도달한 주문은 어떤 상태로도 전이할 수 없음.
