# Milestone 8 — KIS HTTP 실전화 + WebSocket Client + 실시간 루프

## 1. 목적

Milestone 7까지 stub 상태인 KIS 연동을 실제 HTTP/WebSocket 기반으로 전환하고,
실시간 시세/주문체결 이벤트 루프를 구축한다.

## 2. 설계 원칙

1. **Stream A (REST HTTP)를 먼저 수행** — endpoint/TR ID/hashkey/normalizer/error mapping 정합성 우선 확정
2. **WebSocket 이벤트는 source of truth가 아니다** — WS는 append-only ingest + normalized event production까지만 담당
   - 주문 상태 반영은 반드시 OrderManager/ReconciliationService 경로를 통해 처리
3. **SubscriptionBudget은 실제 critical/optional eviction 정책까지 구현**
4. **모든 KIS endpoint/TR ID 수정은 reference_docs 기준 검증 근거를 남길 것**
5. **테스트 우선** — WS disconnect, duplicate event, gap fill, ambiguous state lock, budget fallback, subscription saturation 테스트를 먼저 추가
6. **audit log ordering 테스트 실패 1건** — known issue로 명시, 선해결 또는 우회

## 3. Stream 상세

### Stream A: KIS REST Client HTTP 실전화

**목표**: stub 응답 → 실제 KIS HTTP API 호출

**파일**: [`src/agent_trading/brokers/koreainvestment/rest_client.py`](../src/agent_trading/brokers/koreainvestment/rest_client.py)

**구현 항목**:
- `httpx.AsyncClient` 기반 HTTP transport
- Access token 실제 발급 (POST `/oauth2/tokenP`)
- Approval key 실제 발급 (POST `/oauth2/Approval`)
- 주문/조회/시세 endpoint 실제 호출
- Hashkey 생성 (HMAC-SHA256)
- 공통 header 조립 (authorization, appkey, appsecret, tr_id, custtype)
- 응답 정규화 (rt_cd / msg_c1 → domain model)
- Error mapping (KIS error code → BrokerErrorType)

**Endpoint 오류 수정** (reference_docs 검증 완료):

| 코드 내 endpoint | 수정 후 | 근거 |
|---|---|---|
| `inquire-daily-ccnl` | `inquire-daily-ccld` | 한투 API 문서 v1_국내주식-005 |
| `inquire-psamount` | `inquire-psbl-order` | 한투 API 문서 v1_국내주식-007 |
| `inquire-asking-price-exp` | `inquire-asking-price-exp-ccn` | 한투 API 문서 실시간호가 |
| `order-correct` (불필요) | 제거 | 정정/취소는 `order-rvsecncl` 통합 |
| `order-cc` (불필요) | 제거 | 정정/취소는 `order-rvsecncl` 통합 |
| `inquire-ccnl` | `inquire-ccld` | 체결조회 |
| (누락) `inquire-psbl-rvsecncl` | 추가 | 정정취소가능주문조회 |
| (누락) `inquire-psbl-sell` | 추가 | 매도가능수량조회 |

**TR ID 매핑**:

| Operation | 실전 TR | 모의 TR |
|---|---|---|
| 매수 | TTTC0012U | VTTC0012U |
| 매도 | TTTC0011U | VTTC0011U |
| 정정취소 | TTTC0013U | VTTC0013U |
| 일별주문체결조회 | TTTC0081R | VTTC0081R |
| 잔고조회 | TTTC8434R | VTTC8434R |
| 매수가능조회 | TTTC8908R | VTTC8908R |
| 매도가능수량조회 | TTTC8408R | (모의 미지원) |
| 주식현재가 시세 | FHKST01010100 | FHKST01010100 |

**Error mapping 규칙**:

| KIS 응답 조건 | accepted | uncertain | requires_reconciliation | normalized_status |
|---|---|---|---|---|
| 정상 접수 (ODNO 있음) | True | False | False | ACKNOWLEDGED |
| ODNO 있으나 응답 지연 | True | True | False | ACKNOWLEDGED |
| 응답 없음 / 타임아웃 | False | True | True | RECONCILE_REQUIRED |
| 명시 거절 (rt_cd != "0") | False | False | False | REJECTED |
| HTTP 429 | False | False | True | RECONCILE_REQUIRED |
| HTTP 5xx | False | True | True | RECONCILE_REQUIRED |
| 알 수 없는 오류 코드 | False | True | True | RECONCILE_REQUIRED |

### Stream B: KIS WebSocket Client

**목표**: 실제 WebSocket 연결, 구독, 데이터 파싱, 재연결 복구

**신규 파일**: 
- [`src/agent_trading/brokers/koreainvestment/websocket_client.py`](../src/agent_trading/brokers/koreainvestment/websocket_client.py)
- [`src/agent_trading/brokers/koreainvestment/ws_parser.py`](../src/agent_trading/brokers/koreainvestment/ws_parser.py)

**WebSocket TR ID**:
- `H0STCNT0` — 국내주식 실시간체결가 (KRX)
- `H0STASP0` — 국내주식 실시간호가 (KRX)
- `H0STCNI0` — 국내주식 실시간체결통보 (AES 복호화 필요 가능)

**websocket_client.py 책임**:
- asyncio 기반 WebSocket 연결 관리
- approval_key header 설정
- 연결 lifecycle (connect / disconnect / reconnect with backoff)
- Heartbeat 관리
- Subscription budget 연동 (Stream D)
- Gap detection — 마지막 수신 timestamp 저장

**ws_parser.py 책임**:
- JSON subscription ack 파싱
- `|` 와 `^` 구분자 실시간 데이터 파서
- 다건 패킷 분해 (여러 tick이 한 프레임에 올 수 있음)
- H0STCNI0 AES 복호화 처리 (iv, key 관리)
- 단건 전제 금지 — 반드시 다건 처리

**중요: WS 이벤트는 source of truth가 아니다**
- WS ingest는 append-only로 FillEventEntity / OrderStateEventEntity 저장까지만
- 주문 상태 변경은 OrderManager.transition_to()를 통해서만 반영
- ReconciliationService가 최종 상태 확정

---

### Stream C: 실시간 주문/체결 루프 + Gap Fill

**신규 파일**: [`src/agent_trading/services/event_loop.py`](../src/agent_trading/services/event_loop.py)

**event_loop.py 책임**:
- WS 이벤트 소비 큐
- 체결통보(H0STCNI0) → FillEventEntity 저장
- 체결통보 → OrderRequestEntity 상태 업데이트 (OrderManager 경유)
- REST gap fill: WS 단절 후 inquire-daily-ccld로 누락 구간 조회
- Timestamp 기반 gap detection
- Gap fill 완료 전 해당 종목 fast execution signal stale 표시
- 주문/체결 채널 gap은 market data gap보다 높은 severity

**reconciliation_service.py 보강**:
- WS gap fill 연동
- Gap fill 결과에 따른 reconciliation trigger

---

### Stream D: Subscription Budget 실제 적용

**파일**: [`src/agent_trading/brokers/koreainvestment/websocket_client.py`](../src/agent_trading/brokers/koreainvestment/websocket_client.py)

**구현 정책**:
- 구독 시 `SubscriptionBudget.subscribe_critical()` / `subscribe_optional()` consume
- 해지 시 `SubscriptionBudget.unsubscribe()` release
- Budget 초과 시 **optional부터 eviction**
- Gap fill용 REST 예산은 별도 유지 (MarketData bucket과 분리)

**Critical/Optional 분류**:
- Critical: 보유 종목, 미체결 주문 종목, 진입 후보 상위 N개
- Optional: watchlist 하위 후보, 비보유 관찰 종목

## 5. 테스트 전략

### 선행 테스트 (구현 전 추가)

| 테스트 | Stream | 설명 |
|---|---|---|
| WS disconnect → reconnect → gap fill | B, C | WS 단절 후 재연결 및 누락 데이터 복구 |
| Duplicate event ingestion | B | 동일 체결 이벤트 중복 저장 방지 |
| Ambiguous state lock prevent | A, C | 불명확 상태에서 신규 주문 차단 |
| Budget fallback chain | D | INQUIRY → RECONCILIATION reserve fallback |
| Subscription saturation eviction | D | Budget 초과 시 optional 제거 |
| Gap fill with REST inquiry | C | WS 누락 구간 REST 조회 |

### Known Issue

**audit log ordering 테스트 1건 실패** — 선해결 또는 known issue로 명시.
원인 파악 후 Stream A 진행 전 또는 병행 수정.
